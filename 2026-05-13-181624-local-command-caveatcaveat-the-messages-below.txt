  ---
  1. Target platforms for v1

  The base project currently targets Android, iOS, JS, and WasmJs. Several
  requirements you listed map poorly to the browser:
  - SQLDelight on Wasm/JS is immature (IndexedDB driver exists but with
  limitations).
  - Biometrics and OS-level secure storage (Keychain / Keystore) don't exist in
  browsers.
  - PIN-locking only makes sense for installable apps.

  Suggested default: scope v1 to Android + iOS, drop js and wasmJs targets (or
  keep them as stubs that render an "unsupported platform" screen), and add 
  Desktop (JVM) as an optional third target since it's cheap and aids local
  development/testing. Confirm or override?

  2. Domain model: accounts/wallets?

  You described transactions, categories, income/expense — but not whether
  transactions belong to an account (e.g., Cash, Checking, Credit Card) or are
  bucketed only by category.

  Suggested default: include a lightweight Account concept (Cash + one default),
   so balances and future bank-sync are possible. Single default account hidden
  in UI until the user adds more. Confirm?

  3. Currency handling

  Single currency or multi-currency?

  Suggested default: single currency per user, configurable in settings,
  defaulting to device locale; store amounts as Long (minor units, e.g. cents)
  to avoid float errors. Add a currencyCode field on transactions for
  future-proofing multi-currency, but don't expose FX in v1. Confirm?

  4. PIN + biometrics behavior

  A few sub-questions:
  - Is the PIN mandatory or opt-in?
  - Is biometric a replacement for PIN (PIN as fallback) or an addition (must
  set PIN first, then enable biometric)?
  - Lock on app launch only, or also on background/foreground transition after N
   seconds?

  Suggested default: opt-in via Settings; user sets a PIN first, biometric is an
   optional unlock method that always falls back to PIN; lock on cold launch and
   after 30s in background. Confirm?

  5. Navigation library

  Three credible choices in KMP today:
  - Jetpack Navigation Compose for KMP (now multiplatform-ready, official) —
  simplest, familiar.
  - Decompose — most architecturally rigorous (real component tree, lifecycle,
  state restoration), best for complex flows; steeper learning curve.
  - Voyager — middle ground, batteries-included (tabs, bottom sheets), but
  smaller community.

  Suggested default: Jetpack Navigation Compose (multiplatform) — lowest
  friction, official, sufficient for this app's flow complexity. Confirm or
  prefer Decompose/Voyager?

  6. Dependency injection

  Suggested default: Koin (most popular in KMP, runtime-based, simple).
  Alternative is Kotlin-Inject (compile-time, safer, slightly more ceremony).
  Confirm Koin?

  7. Charts library

  KMP charting is a weak ecosystem. Options:
  - Koalaplot — multiplatform, line/bar/pie, decent but not feature-rich.
  - Compose Charts (e.g. io.github.ehsannarmani:compose-charts) — multiplatform,
   more chart types, less mature.
  - Roll our own with Canvas for the simple shapes (bar/line/donut) — most
  control, more work.

  Suggested default: Koalaplot for v1; isolate it behind a charts module so we
  can swap it. Confirm?

  8. Date/time, serialization, settings

  I plan to use kotlinx-datetime, kotlinx-serialization (JSON + CSV via a thin
  layer), multiplatform-settings (russhwolf), and kotlinx-coroutines. These are
  the de-facto standards. Suggested default: use these unless you object.

  9. Recurring transactions in v1?

  You didn't mention recurring/scheduled transactions. They affect the data
  model.

  Suggested default: out of scope for v1, but the schema reserves room for it
  (recurrence_rule nullable column). Confirm?

  10. Attachments (receipt photos)?

  Out of scope for v1 unless you say otherwise. Suggested default: out of scope.

  11. Onboarding & first-launch experience

  Suggested default: brief 1–2 screen onboarding on first launch — pick
  currency, optionally set PIN, optionally enable biometric. Skippable. Confirm?

  12. App identity

  - Project directory is MoneyM2, Gradle rootProject.name = "MoneyM", Android
  namespace com.dv.moneym. Keep com.dv.moneym as the package across modules? Any
   preferred display name (e.g., "MoneyM2", "Money", custom)?

  Suggested default: keep package com.dv.moneym, display name "MoneyM2".
  Confirm?

  13. Localization scope

  Suggested default: English only for v1; all user-facing strings go through
  Compose resources so localization is additive later. Confirm?

  14. Testing depth for v1

  Suggested default: unit tests for ViewModels + UseCases + Repositories
  (Turbine for Flow assertions, Kotest assertions or kotlin.test); a small set
  of Compose UI tests for critical screens; no end-to-end tests in v1. Confirm?

  Risks and tradeoffs (full list in docs/architecture/decisions.md)

  - Koin runtime DI — missing bindings fail at runtime, not compile time.
  Mitigation: a DI-resolution test in composeApp:commonTest that instantiates
  every ViewModel.
  - No DB encryption in v1 — we rely on OS sandbox encryption. A user without a
  device passcode is exposed under physical-extraction threat. Stated in
  security.md; SQLCipher is a future option.
  - Argon2id on iOS — binary-size cost is unmeasured. ADR-010 is open; PBKDF2
  with 600k iterations is the universal fallback.
  - Compose Charts — KMP charting is immature; the library is wrapped per chart
  type so a swap costs one module.
  - Feature-first modularity — more build.gradle.kts files to keep aligned. If
  the overhead bites, we add convention plugins after Phase 3.
  - MockK skipped — Mockative + hand-rolled fakes cover everything but with a
  different API; the testing skill makes the rules clear so new tests don't
  drift back to MockK.
